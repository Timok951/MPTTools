Резервное копирование и восстановление PostgreSQL

Для PostgreSQL используйте логический дамп и восстановление через стандартные утилиты. Это удобно показать на защите как регламент восстановления.

Резервная копия

pg_dump -h {{ db_host|default:"localhost" }} -p {{ db_port|default:"5432" }} -U {{ db_user|default:"postgres" }} -d {{ db_name }} -Fc -f backup.dump

Восстановление

pg_restore -h {{ db_host|default:"localhost" }} -p {{ db_port|default:"5432" }} -U {{ db_user|default:"postgres" }} -d {{ db_name }} --clean --if-exists backup.dump

После восстановления проверьте вход в систему, список оборудования, API `/api/v1/schema/` и запуск одной админской процедуры из портала.

SQL-объекты PostgreSQL

Если нужно вручную прогнать объекты БД в PostgreSQL, используйте файл db/postgresql/001_inventory_objects.sql. В нём лежат views, процедуры и audit triggers.

psql -h {{ db_host|default:"localhost" }} -p {{ db_port|default:"5432" }} -U {{ db_user|default:"postgres" }} -d {{ db_name }} -f db/postgresql/001_inventory_objects.sql

Автоматический серверный бэкап

Для сервера можно настроить ежедневный автоматический дамп PostgreSQL через Docker-контейнер с cron, без привязки к Windows.

Команда Django

.venv\Scripts\python.exe TIP\manage.py create_server_backup --label nightly

Переменные окружения

BACKUP_KEEP_COUNT=14
BACKUP_CRON_SCHEDULE=0 2 * * *

Docker Compose

docker compose -f docker-compose.backup.yml up -d --build

Логи контейнера

docker compose logs -f backup

Файлы Docker-автобэкапа лежат в docker-compose.backup.yml и папке docker/backup/. Дампы сохраняются в backups/postgresql.
